# Campaign 03 — H100 narrow-replication paired-control (2026-04-27)

**Status:** PLANNED

Campaign 03 turns Campaign 02's single suggestive datapoint into
publishable evidence by:

1. **Replicating** the +14.7% rmsnorm/fp16/large NOV win (and the
   −11.13% rmsnorm/bf16/medium NOV loss, and a tie cell) at N=3 paired
   observations each. Campaign 02 had N=1 per cell which can't
   distinguish a real LLM-novel win from "the LLM happened to emit a
   good kernel that one time" — replication is the only honest answer.
2. **Running on H100** (1× H100 campaign container) instead of A100.
   Campaign 02's L3 trials all ran on A100; whether the LLM-novel
   kernel wins on H100 too is the hardware-class generality question.
   Combined with Campaign 02's same-cell A100 data, Campaign 03 gives
   the first cross-hardware paired evidence for the kernel-novelty
   thesis claim.
3. **Tightening the paired-prompt path** via T-29: each cell gets its
   own LLM round-trip instead of asking for all blocks in one
   response. Campaign 02's 6-block paired prompt broke 2/6 NOV halves
   (33% vs ≤17% target), all in the silu_mul block — late-block
   degradation. Per-cell calls eliminate that coupling.

T-26b (per-knob feature weights) ships as pre-flight, addressing
Campaign 02's Q2 negative result.

## Goal — questions to answer

| # | Question | Mechanism | Success criterion |
|---|---|---|---|
| **Q1** | Does the rmsnorm/fp16/large NOV +14.7% from Campaign 02 reproduce on H100 with N=3? | 3 paired observations at the cell, each with a fresh LLM-novel kernel emission. Aggregate as win-rate (NOV beats REF by ≥3%). | ≥ 2/3 NOV wins (mean Δ% > 0 with no NOV loss > 5%) → robust replication. 0–1/3 NOV wins → the C02 +14.7% was lucky-emission noise. The win-rate, not just the mean Δ%, is the citable outcome. |
| **Q2** | Does T-26b's per-knob feature weighting raise L1 surrogate kept-rate above C02's 8.3%? | Class-aware feasibility classifier with `kv_cache_dtype` and other catalog-rule knobs upweighted 10x. On H100 the FP8 dead zone disappears, but the classifier should still filter `fp8 + XFORMERS` and other catalog-violation regions. | L1 surrogate kept-rate ≥ 30% (target from C02). On H100 the FP8 region is feasible (sm_90), so failure modes are narrower — the test isn't about FP8 suppression, it's about the classifier learning whatever H100-specific failure modes do exist. |
| **Q3** | Cross-hardware kernel-novelty: do C02 (A100) and C03 (H100) paired results agree on which cells are NOV-favorable? | Compare Δ% per cell between C02 (A100, N=1) and C03 (H100, N=3). | Agreement (within ±5% Δ% per cell) → kernel-novelty generalizes across A100 → H100 at this surface. Disagreement → hardware-class-specific kernel wins; widens the methodology footprint. |
| **Q4** | Does T-29's per-cell LLM call protocol reduce paired-prompt parser brittleness from C02's 33% to under 17%? | Each cell's LLM call is independent — silu_mul-side breakage cannot propagate from rmsnorm-side breakage. | NOV-half failure rate < 1/9 (≤ 11%) over the 9 paired observations. Above that → the LLM is genuinely struggling with kernel emission, not delimiter hygiene; opens T-30 for kernel-emission-specific fine-tuning or richer few-shot. |

Q1 is highest-value (turns C02's suggestive datapoint into citable
evidence). Q3 is the cross-hardware claim that makes the framework
publishable. Q2 and Q4 are derivative methodology checks.

## Methodology — how each Q gets measured

### Q1 — paired same-cell replication

For each of the 3 cells (rmsnorm/fp16/large, rmsnorm/bf16/medium,
silu_mul/fp16/large), the paired-control warmstart emits:

```
ref_0 → novel_0 → ref_1 → novel_1 → ref_2 → novel_2
```

= 6 trials per cell × 3 cells = 18 paired-control warmstart trials.
Each `novel_i` is a fresh LLM-novel kernel emission via T-29's
per-cell call (so emission variance across the 3 replicates is real
LLM variance, not the same kernel re-run).

The post-run analysis groups by cell, computes per-trial Δ% NOV-vs-REF
within each (ref_i, novel_i) pair, and reports:

| Cell | n_pairs | win-rate | mean Δ% | Δ% range | NOV KL range |
|---|---|---|---|---|---|
| rmsnorm/fp16/large | 3 | …/3 | … | … to … | … to … |
| rmsnorm/bf16/medium | 3 | …/3 | … | … to … | … to … |
| silu_mul/fp16/large | 3 | …/3 | … | … to … | … to … |

A "win" = NOV tok/s ≥ REF tok/s × 1.03 with NOV KL within `gate.max_kl=5.0`.

The cleanest publishable result is: "in 3/3 trials at rmsnorm/fp16/large
the LLM-novel kernel beat the reference by 8–18%." Anything noisier
(e.g. 1/3 wins with high variance) is also publishable but lands as
"effect not robust at N=3, more replication needed."

### Q2 — class-aware feasibility on H100

H100 has no FP8 dead zone, so the classifier doesn't get the
FP8-on-A100 free lunch. The remaining failure modes the classifier
should learn are:
- FP8 + XFORMERS (catalog rule, hardware-independent)
- OOM regions when the H100's 80GB hits gpu_memory_utilization × max_num_seqs
- QUALITY_KL drift on aggressive quantization combos

Measurement:
```
n_warmstart_kept / n_warmstart   = … / 4
n_operator_kept   / n_operator    = … / ~3
n_surrogate_kept  / n_surrogate   = … / ~7
n_reserve_kept    / n_reserve     = … / 0–6  (only fires if L2 stale-fires)
```

Headline: L1 surrogate kept-rate. Target ≥ 30%.

Auxiliary: did the classifier filter any catalog-violation proposals
before subprocess spawn? Pre-spawn rejects appear as `elapsed_s ≈ 1e-4`
in trial JSONs.

### Q3 — cross-hardware comparison

Build a per-cell delta table from C02 (A100, N=1) and C03 (H100, N=3):

| Cell | C02 NOV Δ% (A100, N=1) | C03 mean NOV Δ% (H100, N=3) | Agreement? |
|---|---|---|---|
| rmsnorm/fp16/large | +14.7% | … | … |
| rmsnorm/bf16/medium | −11.13% | … | … |
| silu_mul/fp16/large | +0.77% | … | … |

"Agreement" criterion: signs match AND magnitudes within ±5%. The
campaign cannot answer "do these specific kernels win on a third
hardware class" — that's a future replication. But A100 ↔ H100
agreement at three cells is enough to claim "the LLM-novel kernel's
edge generalizes within the same SM-architecture family."

If the signs **don't** match (e.g. NOV beats REF on A100 but loses on
H100), that's an honestly-publishable nuanced result — kernel novelty
is hardware-class-specific, the methodology footprint is narrower
than the Q1-only writeup would suggest. We'd expand the kernel
proposer's hardware-aware prompting in a follow-up.

### Q4 — T-29 parser reliability

Count `[autoinfer.l3.proposer] paired-control fallback` markers in
the campaign-runner stdout. Target ≤ 1 over the 9 paired observations
(was 2 over 6 in C02). Each marker means an LLM call returned no
parseable block; the cell falls back to its reference (the pair
degenerates to two reference trials at the same cell — still useful
data but doesn't add a NOV observation).

Auxiliary: per-cell parser reliability. If silu_mul still has a
disproportionate fallback rate even with per-cell calls, the LLM is
struggling with silu_mul kernel emission generally, not delimiter
hygiene; opens T-30 for richer silu_mul few-shots or fine-tuning.

## Expected outcomes

### Outcome A — Q1 robust replication at the C02 winner
*Probability:* medium-low

What we'd see at rmsnorm/fp16/large: 3/3 NOV wins with mean Δ% in
+10% to +20% range. The C02 +14.7% is not noise; the LLM-novel rmsnorm
kernel for fp16/large is genuinely better than the PyTorch reference
on H100.

What it answers: Q1 robustly affirmed. Combined with C02's A100
datapoint (+14.7%), Q3 likely also agrees. **The article-grade headline
result.** Triggers a Campaign 04 with replication on a third hardware
class (RTX A6000 or A100-40GB) for full hardware-class coverage.

### Outcome B — Q1 mixed (1/3 or 2/3 wins, high variance)
*Probability:* medium-high

What we'd see at rmsnorm/fp16/large: e.g. one trial +18%, one trial
+2%, one trial −5%. The C02 +14.7% was real but the LLM-novel kernel
emission has high variance — sometimes it produces a strong winner,
sometimes barely a tie.

What it answers: Q1 partially affirmed. Worth publishing as "in 1-of-3
to 2-of-3 trials, the LLM-novel rmsnorm kernel for fp16/large beats
the reference by N%; effect size is real but emission variance is
high." Opens T-31 for kernel-emission-quality measurement (perplexity-
style scoring of the LLM's kernel proposals before serving).

### Outcome C — Q1 negative (0/3 NOV wins)
*Probability:* low-medium

What we'd see: all 3 NOV trials within ±2% of REF, or all worse. The
C02 +14.7% was lucky single-shot emission noise.

What it answers: Q1 honestly negative at this cell. The LLM-novel
kernel architecture works (correctness gate passes) but doesn't
reliably outperform vLLM's reference on rmsnorm/fp16/large. Could
still be cell-specific (some cells genuinely have headroom; others
don't). Triggers Campaign 04 either expanding to more cells or moving
to T-21 (attention injector).

### Outcome D — Q2 affirmed on H100
*Probability:* medium

What we'd see: L1 surrogate kept-rate ≥ 30%. The classifier learns
H100-specific failure modes (FP8 + XFORMERS, OOM regions) within the
12-trial budget.

### Outcome E — Q2 partial / negative on H100
*Probability:* medium

What we'd see: surrogate kept-rate 15-25%. T-26b's weights help
versus pure T-26 but don't clear 30%. Likely cause: H100 has fewer
deterministic-failure regions than A100 (no FP8 dead zone), so the
classifier has less obvious patterns to learn — most failures are
softer (OOM, KL drift) and don't generalize cleanly through the k-NN
distance.

If this lands, opens T-26c for per-FailureKind sub-classifiers OR a
finer-grained surrogate-feedback loop.

### Outcome F — Q3 cross-hardware agreement
*Probability:* medium

C02 (A100) and C03 (H100) deltas agree in sign and within ±5% magnitude
at all 3 cells. **Strongest possible cross-hardware claim** at this
N. Article-grade.

### Outcome G — Q3 cross-hardware disagreement
*Probability:* medium

Some cells flip sign between A100 and C03 H100. Honest publishable
result with a narrower scope claim ("kernel novelty is hardware-
class-specific at these cells"). Triggers a follow-up campaign with
hardware-aware kernel prompting (LLM gets told `target_hw = H100`
explicitly).

### Outcome H — T-29 fixes parser brittleness
*Probability:* high

Sequential per-cell calls drop NOV-half failure rate from C02's 33%
to under 11%. silu_mul side no longer disproportionately broken.

### Outcome I — Integration breakage
*Probability:* low (T-26b + T-29 + paired-control extensions all
ship with full test coverage; 334 tests pass + ruff + mypy clean).

What we'd see: an unexpected interaction, e.g. T-26b's weights make
the classifier reject ALL surrogate proposals because catalog rules
on H100 produce different boundary patterns than A100. Diagnosis from
events.jsonl + per-trial JSONs.

What it answers: a different question. Fix and re-run. Budget tolerance:
1 retry within $30.

## Decision tree from the data

```
If Q1 robust replication (Outcome A) AND Q3 agreement (Outcome F):
    → Article-grade headline. Campaign 04 = third hardware class
      (RTX A6000) for full hardware coverage.
    → Update TODO.md: T-26-followup closes; T-31 (replication on
      third hardware) opens; T-21 stays deferred.

If Q1 mixed (Outcome B) regardless of Q3:
    → "Effect is real but emission variance is high."
    → Open T-31: kernel-emission-quality scoring.
    → Campaign 04 considers more cells × more N to nail down which
      cells robustly favour LLM-novel.

If Q1 negative (Outcome C):
    → Architecture works at correctness, doesn't reliably win on
      rmsnorm/fp16/large.
    → T-21 (attention injector) escalates to in-progress; that's the
      next surface where measurable kernel wins are most likely.
    → Open T-32 for richer kernel-emission methodology (multi-shot
      LLM ensembles, automated-kernel-search variants).

If Q2 affirmed (Outcome D):
    → T-26b validated end-to-end. Update TODO: T-15 (retire hardware-
      notes prose) becomes data-actionable. Trim the FP8-on-A100
      hand-authored notes since the classifier demonstrably learns
      this.

If Q2 partial / negative (Outcome E):
    → Open T-26c: per-FailureKind sub-classifiers.

If Q3 disagreement (Outcome G):
    → Open T-33: hardware-aware kernel prompting (LLM gets target_hw
      explicitly).

If T-29 brittleness persists (NOT Outcome H):
    → Open T-30: silu_mul-specific few-shots OR fine-tuned proposer.

If integration breakage (Outcome I):
    → Diagnose from events.jsonl + per-trial _vllm.{out,err}.
    → Fix and re-run; budget tolerance: 1 retry within $30.
```

## What this campaign does NOT yet show

Honest scope. Things we explicitly are NOT trying to answer in this run:

- **Third hardware class.** Only A100 (C02) and H100 (C03) covered.
  RTX A6000 / A100-40GB / B200 require Campaign 04+.
- **Other models.** Qwen3-8B only. Other models with different
  layer-share profiles (e.g. attention-light Llama 70B) require
  separate campaigns.
- **Other cells.** silu_mul/bf16/medium and the other 2 C02 cells we
  dropped aren't in the C03 cell list; their replication is a future
  campaign if needed.
- **L3 wins on the hot path.** Attention is ~70% of Qwen3-8B's compute;
  rmsnorm + silu_mul together ~5–10%. T-21 covers attention; this
  campaign does not.
- **Production-traffic patterns.** ShareGPT-64-prompt synthetic load
  same as C01/C02. Real-traffic shape variance is out of scope.
- **N>3 per cell.** N=3 is the minimum for honest win-rate. N=10+
  for tight confidence intervals is a future replication study.
